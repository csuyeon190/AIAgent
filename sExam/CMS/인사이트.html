3. 이벤트 바인딩 (문서 준비 후) • 셀렉트박스 change.customSelect → 칩 다시 렌더링 • 칩의 삭제 버튼 클릭 → data-select-name으로 해당 셀렉트박스를 찾아 CustomSelectbox.unselect() 호출. 이러면 change 이벤트가 다시 돌면서 칩도 갱신되는 구조입니다 (이벤트 위임 $(document).on("click", ...)을 쓴 건 칩이 동적 생성되기 때문) • 리셋 버튼 → 키워드 입력값 비우기 + 모든 셀렉트박스 초기화 + 칩 영역 비우기 + 포커스 상태 해제 4. 검색창 열기/닫기 로직 (가장 정교한 부분) 포커스 시 is-focus 클래스를 붙이고, 모바일(layout-mo)이면 NonScroll.enable()로 배경 스크롤을 잠급니다. 닫는 경로가 세 가지인데: • 바깥 클릭: click.searchForm에서 클릭 대상이 검색창 내부가 아니면 닫기 • Tab 키로 포커스 이탈: isTabKey 플래그가 핵심입니다. focusout은 마우스 클릭으로도 발생하는데, 클릭 케이스는 위의 click 핸들러가 처리하므로 중복을 피하려고 “직전 입력이 Tab이었을 때만” focusout에서 닫습니다. e.relatedTarget(다음 포커스 대상)이 검색창 내부면 아직 안에 있는 것이니 닫지 않고요 • 검색 버튼 / 키워드 버튼 클릭: 명시적으로 closeSearchForm() 호출. 키워드 버튼은 추가로 해당 키워드를 input에 넣고 trigger("input")으로 입력 이벤트를 발생시킵니다 한 가지 눈에 띄는 점: closeSearchForm($searchForm)처럼 인자를 넘기는 곳이 있지만 함수 정의는 인자를 받지 않아서 무시됩니다(동작엔 문제없음). 그리고 칩 렌더링 시 text를 템플릿 리터럴에 그대로 넣고 있어서, 옵션 텍스트가 서버/사용자 입력에서 온다면 XSS 관점에서 이스케이프를 고려할 만합니다 — 지금 하고 계신 DOMPurify 하드닝 작업 범위에 포함시켜도 좋겠네요.
